## Overview

"20CC" is the proprietary SID replay routine of the Dutch group **20th Century
Composers** (20CC), developed by **Falco Paul** (co-founder with Edwin van
Santen, or EVS) starting in late 1986. It was never a publicly released editor —
it was the group's in-house SID player technology, embedded in their compositions
and lent to trusted collaborators like Reyn Ouwehand. Falco Paul worked on it
continuously through ~1991, refining multispeed playback, sample support, PWM,
and his signature "auto-swing" feature.

242 files across 30 composers survive in HVSC, concentrated in the Dutch scene
(EVS alone accounts for 28%). The routine exists in at least 4 versions (3 v1
variants + v2) per PRG2SID, and a separate variant was made for Mitch/Crest by
Mark (Apache) in 1993/1994. The original Turbo Assembler source reportedly
survives on untransferred floppy disks.

Some RSID rips are IRQ-driven (play=$0000); the 3 PSID files verified here call
play directly via a non-zero PSID play address and did not need IRQ emulation.
**As of 2026-07-23, this player has been disassembled and register-write-exact
verified on 3 real HVSC files** (see Verification section): a fixed 3-byte
init routine, a $F9-$FF zero-page footprint, and a per-voice parameter table
right after the load address are now confirmed facts, not TODOs. The full
data/pattern format and effect-command encoding remain TODO — verification
here closed the entry-point/ZP/byte-reconstruction gap, not the musicological
data-format walk described in the playbook's steps 3-5.

## Quirks & gotchas

See the `quirks` array. The load-bearing ones: it is a **group-internal
routine, not a public tool**; **CSDb 10741 is a third-party wrapper, not the
real editor** (confirmed by CSDb commenter Fred, 2014); there is a
**founding-date conflict** (late 1986 primary source vs mid-1990s secondary
claim); the player is **IRQ-driven** (RSID play=$0000, bare init+play trace
won't work); the string "20CC" in unrelated SID titles (e.g. Rokling's
"20CC-Shit") is coincidental.

## Disassembly notes

Three real HVSC files disassembled with `SIDdecompiler.exe -a<decimal load
addr> -z -d -c -v2` (decimal load address computed from each file's own PSID
header, per this project's `psid_header` convention — all 3 files have
`loadAddr0=0`, so the real load address is the payload's own first 2 LE
bytes), reassembled with `64tass.exe -a --cbm-prg`.

- **`SIDdecompiler`'s `-v2` `Start:`/`End:` addresses matched each file's own
  PSID load address exactly** in all 3 cases — no relocation-mismatch
  (gotcha 40) trap hit here.
- **A recurring assembly blocker**: SIDdecompiler cannot encode the
  undocumented 6502 opcode `$1A` and emits a bare `ill` line (0 bytes) —
  hit on 2 of 3 files (Hip_House.sid at `$11f8`, Amtrak.sid at `$11b2`),
  silently shortening the reassembly by 1 byte and cascading a large,
  misleading byte-diff (~17-20% match) until fixed. See quirks.
- **After that fix, all 3 files byte-diffed to 97.6-98.4% match**,
  concentrated entirely in one contiguous region right after the load
  address (a per-voice parameter table) — every diverging address is
  `-v2`-map `+`/`w`/`_`/`B` marked (write-touched). Patched those bytes
  back to the real file's own pristine values (not guessed — read directly
  from the payload) to close to 100.0000% byte-exact on all 3 files.
- Confirmed via trace-diff (not assumed, per the project's own precedent
  for this class of gap) that the pristine table values are load-bearing,
  not dead: an unpatched Hip_House.sid reassembly diverges from the
  original SID-register trace starting frame 1 (wrong/extra pulse-width
  and oscillator-frequency writes); the patched version matches exactly.
- Play/init entry-point code, and the $F9-$FF zero-page footprint, are now
  documented facts — see `entry`/`memory.zero_page` in the JSON block.
- Data pattern/order-list format, instrument/wavetable structure, and
  effect-command encoding remain undocumented (playbook steps 3-5) — this
  pass closed the entry-point/reconstruction gap, not the musicological
  walk. The IRQ-driven RSID path (play=$0000, e.g. `Greystorm.sid`), the
  PRG2SID-documented "v2" build (init at `$0ffb`, fixed play `$1000`), and
  the Mark/Apache Mitch/Crest variant remain untested — no file matching
  those specific signatures was located/tried in this pass.

## Verification

**`status: verified` (2026-07-23) — a register-write-exact reconstruction
confirmed on 3 independent real HVSC files**, all sharing the same core
init/play code (see Disassembly notes): `Hip_House.sid` (root 20CC folder,
credited "Edwin van Santen & Falco Paul"), `van_Santen_Edwin/13_Seconds_of_Massacre.sid`,
and `van_Santen_Edwin/Amtrak.sid`.

Method, per `knowledge/playbooks/disassemble-a-player.md` and this project's
`sid-player-verify` process:

1. Read each file's real PSID header (load/init/play addresses) directly —
   all 3 have `loadAddr0=0` (load address embedded in the payload's own
   first 2 bytes).
2. Disassembled with `SIDdecompiler.exe -a<decimal load> -z -d -c -v2`;
   confirmed the `-v2` map's `Start:` address matches the PSID load address
   in every case (no relocation trap).
3. Reassembled with `64tass.exe`. 2 of 3 files needed a 1-byte fix first
   (SIDdecompiler's `ill` placeholder for undocumented opcode `$1A` — see
   quirks) before the reassembled length matched the original payload
   length.
4. Byte-diffed the reassembled `.prg` against the pristine original SID
   payload: **Hip_House.sid — 79/3,273 bytes differed** (addr
   `$1021-$10d2`), **13_Seconds_of_Massacre.sid — 46/2,819 bytes differed**
   (addr `$103c-$10c3`), **Amtrak.sid — 68/3,064 bytes differed** (addr
   `$1000-$106f`, plus 3 scattered single bytes at `$124d`/`$1697`/`$16b0`).
   Every diverging address is `-v2`-map write-touched (`+`/`w`/`_`/`B`).
5. Patched the reassembled `.prg` binary directly at every diverging
   address, byte-for-byte, to the real file's own pristine value (not the
   decompiler's post-execution snapshot) — all 3 files reached
   **0 remaining byte differences (100.0000% byte-exact)**.
6. Traced both the original and patched `.prg` via
   `sidm2-sid-trace.exe <prg> 300 <init_hex> <play_hex> 0` (300 frames,
   each file's own real PSID init/play addresses) and diffed the two
   register-write logs: **all 3 files — 0 register-write differences over
   300 frames** (only the tool's own echoed input filename line differs).
   Before patching, the same trace-diff on Hip_House.sid genuinely
   diverged starting frame 1 — confirming the drifted table was
   load-bearing, not dead workspace, per this project's own precedent for
   this class of gap (never assume a write-touched byte is dead without
   trace-diff confirmation).

**Scope honestly stated**: this verifies the entry-point/ZP/byte-level
reconstruction for the "v1-style, direct PSID play address" build confirmed
identical across all 3 files tested (Falco Paul's own credit appears in one
of them, and EVS — the group's primary composer — in the other two). It does
**not** cover: the IRQ-driven RSID path (play=$0000), the PRG2SID-documented
"v2" build (init at `$0ffb`, fixed play at `$1000` — no such file located in
this pass), or the separate Mark/Apache variant made for Mitch/Crest. The
full musicological data format (order list, patterns, instruments,
wave/pulse/filter tables, effect-command encoding) also remains TODO — this
pass closed the reconstruction/entry-point gap the card previously flagged
as blocking `verified`, not the full playbook walk.
